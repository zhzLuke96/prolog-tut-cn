# DCG — 定从句文法

> DCG 描述一个**序列**。操作上，DCG 可用于解析、生成、补全和检查列表形式的序列。

> 前置：[回溯与控制流](c3c_backtracking_control.md) | 难度：★★★ | 后续：[元编译器](c5.md)

DCG（Definite Clause Grammar）是 Prolog 内置的语法设施——`-->` 在编译期展开为带两个差量参数的普通谓词。换句话说，DCG 是一种**受限的宏系统**（见"DCG 与宏"节），专门用于描述序列。

> 类比 | 前端开发者：DCG ≈ parser combinator（PEG.js / nearley）— 都是组合子描述序列。
> 类比 | 后端开发者：DCG ≈ BNF 语法定义— 非终结符和产生式规则。

## 从自然语言说起

前面"从零开始"第 16 章接触过 DCG，当时用它解析简单句子。历史上 Prolog 源于自然语言处理系统，DCG 是这一起源留下的最重要遗产。

看最基本的例子：

```prolog
%% 语法：句子 -> 名词短语 + 动词短语
s --> np, vp.

np --> [the], [cat].
vp --> [sat], [on], [the], [mat].
```

保存为 `dcg_basic.pl`，加载后查询：

```prolog
?- phrase(s, [the, cat, sat, on, the, mat]).
true .
```

> **Prolog 时刻** — DCG 能用同一套规则做解析和生成。给规则，剩下的 Prolog 搞定。

`phrase/2` 是 DCG 的入口：第一个参数是语法规则名，第二个是待解析的列表。

## 前置知识：差量列表

差量列表（Difference List）技巧允许在常量时间内拼接列表。形式上写作 `[1,2,3|T]-T`。

精确地说，这并不是一种特殊的列表。完全普通的列表或部分列表用作额外参数，是这些参数之间的**差**在起作用。附加参数是普通列表，是它们的差值得关注。

### X-Y 模式

一个普通列表 `[1,2,3]` 可以表示为差量列表 `[1,2,3|T]-T`，其中 `T` 是未绑定的尾部变量。差量列表的核心就是一对 `(List, Tail)`，表示"从 List 到 Tail 这段内容"。

```prolog
?- X = [1,2,3|T]-T.
X = [1,2,3|T]-T.
```

### O(1) 拼接

差量列表的拼接是常数时间：

```prolog
append_dl(X-Y, Y-Z, X-Z).
```

对比常规 append——普通 append 需要遍历整个第一个列表，O(n)。差量列表版本直接通过变量绑定完成拼接：

```prolog
?- A = [1,2|T1]-T1, B = [3,4|T2]-T2, append_dl(A, B, C), C = X-[].
A = [1,2,3,4|_]-[3,4|_],   % 简化输出
B = [3,4|_]-_,
C = [1,2,3,4]-[].
```

### flatten 对比

常规 flatten：

```prolog
flatten([], []).
flatten([H|T], Flat) :-
    flatten(H, FH),
    flatten(T, FT),
    append(FH, FT, Flat).
flatten(X, [X]).
```

差量列表版：

```prolog
flatten_dl([], T-T).
flatten_dl([H|T], X-Z) :-
    flatten_dl(H, X-Y),
    flatten_dl(T, Y-Z).
flatten_dl(X, [X|T]-T).
```

查询：

```prolog
?- flatten([1,[2,3],[4]], F).
F = [1,2,3,4].

?- flatten_dl([1,[2,3],[4]], []-[]).
true .
```

### 与 DCG 的联系

DCG 的两个隐藏参数本质上就是差量列表：`s(S0, S)` 表示"从 S0 到 S 这一段内容"。`phrase/2` 调用时传入完整列表，期望最终剩余为 `[]`——这等价于差量列表的 `List-[]` 模式。

```prolog
%% DCG 规则
s --> np, vp.

%% 展开后——差量列表传递
s(S0, S) :- np(S0, S1), vp(S1, S).
%%    S0 是输入列表，S 是剩余列表
%%    中间结果通过 S1 串联
```

理解差量列表，DCG 的隐藏参数就不再"隐藏"了——你看到的 `S0, S1, S` 只是显式串联的差量列表拼接。

## DCG 的本质

每个 DCG 规则编译为带两个额外参数的 Prolog 谓词。这两个参数是"差量列表"（严格说是"列表的差"）。例如：

```prolog
s --> np, vp.
```

编译后等价于：

```prolog
s(S0, S) :- np(S0, S1), vp(S1, S).
```

两个隐藏参数代表"差量列表"——`S0` 是剩余输入，`S` 是消耗后的剩余。`phrase/2` 只是把完整的列表传进去，期望最终剩余为空列表。

终端符 `[word]` 编译为：

```prolog
[word] --> [word].
% 等价于：
[word](S0, S) :- S0 = [word|S].
```

## 实战 1：解析 CSV

```prolog
%% 解析一行 CSV：字段间用逗号分隔
csv_line(Fields) --> field(Fields).

field([F|Fs]) --> value(F), separator_or_end, field(Fs).
field([])     --> [].

value(V) --> token(V).

separator_or_end --> [','].
separator_or_end --> [].

%% 单个 token（不含逗号的连续字符列表）
token([C|Cs]) --> [C], { C \= 0'' }, token(Cs).
token([])     --> [].
```

测试：

```prolog
?- phrase(csv_line(F), "a,b,c").
F = [[97], [98], [99]] .
```

输出是 ASCII 码列表，可以用 `atom_codes/2` 转成原子。

```prolog
?- phrase(csv_line(F), "a,b,c"), maplist(atom_codes, Atoms, F).
F = [[97], [98], [99]],
Atoms = [a, b, c].
```

## 实战 2：简单算术表达式

DCG 天然适合解析嵌套结构。看这个四则运算解析器：

```prolog
expr(E)   --> term(T), expr_rest(T, E).
expr_rest(T0, E) --> [+], term(T), expr_rest(T0+T, E).
expr_rest(T0, E) --> [-], term(T), expr_rest(T0-T, E).
expr_rest(E, E)  --> [].

term(T)   --> factor(F), term_rest(F, T).
term_rest(F0, T) --> [*], factor(F), term_rest(F0*F, T).
term_rest(F0, T) --> [/], factor(F), term_rest(F0//F, T).
term_rest(T, T)  --> [].

factor(N) --> [N], { number(N) }.
factor(E) --> ['('], expr(E), [')'].
```

查询：

```prolog
?- phrase(expr(E), [1,+,2,*,'(',3,+,4,')']), V is E.
E = 1+2*(3+4),
V = 15 .
```

注意这里用 `is/2` 求值——DCG 建的语法树可以延迟计算。

## DCG 与宏

DCG 的本质是编译期语法变换——`-->` 在 Prolog 代码加载前被展开成带两个差量列表参数的普通谓词。这种"写 DSL，生成普通代码"的思路，在其他语言里叫**宏**。

### C 的宏（#define）

纯文本替换，在预处理阶段展开。危险但高效——没有类型检查，参数可能被多次求值：

```c
#define SQUARE(x) ((x)*(x))
int y = SQUARE(++a); // 展开为 ((++a)*(++a)) —— 未定义行为
```

### Lisp/Scheme 的宏

操作 AST 而非文本——模式匹配 + 代码重写，比 C 安全但更灵活：

```scheme
(define-syntax my-when
  (syntax-rules ()
    ((my-when test body ...)
     (if test (begin body ...)))))
```

宏展开在编译期完成，展开后的代码和手写的一样高效。

### Rust 的声明宏（macro_rules!）

模式匹配式语法变换，类似 Lisp 但写起来更像 DCG 规则：

```rust
macro_rules! vec {
    ( $( $x:expr ),* ) => {
        {
            let mut v = Vec::new();
            $( v.push($x); )*
            v
        }
    };
}
```

### 对比

| 特性 | DCG | C 宏 | Lisp 宏 | Rust 宏 |
|------|-----|------|---------|---------|
| 作用域 | DCG 规则内 | 全局文本 | 代码 AST | 模式匹配 |
| 安全性 | 高（语义明确） | 低（文本替换） | 中高 | 中高 |
| 展开时机 | 编译期 | 预处理 | 编译期 | 编译期 |
| 表达能力 | 受限（差量列表） | 任意文本 | 任意 AST | 有限 AST |
| 运行时开销 | 零 | 零 | 零 | 零 |

DCG 是 Prolog 特有的宏系统——受限但有明确语义（差量列表上下文），不像 C 宏那样危险，不像 Lisp 宏那样通用。它在"写语法规则"这个特定场景下，比任何其他语言的宏都更自然。

## DCG debug 技巧

DCG 编译后的隐藏参数让调试有点棘手。几个常用方法：

**1. 用 `listing/1` 看展开结果**

```prolog
?- listing(s/4).
s(A, B) :- np(A, C), vp(C, B).
```

**2. 加 debug 打印（用 `{}` 包装普通谓词）**

```prolog
s --> np, { writeln(parsed_np) }, vp.
```

花括号里的代码是普通 Prolog，不参与差量列表展开。

**3. `phrase/3` 看剩余**

```prolog
?- phrase(s, [the, cat, sleeps, and, dog, barks], Rest).
```

`Rest` 会显示未被消费的部分，帮你定位哪里匹配失败。

**4. `portray/1` 显示差量列表**

在 swipl 的 `~/.swiplrc` 中加入：

```prolog
:- portray(dcg_difflist(_)).
```

## 总结

DCG 是 Prolog 最强大的语法糖之一。它把解析和生成统一到同一套规则里——同一个 DCG 规则既能解析也能生成句子。记住三点：

- `-->` 编译为带两个差量列表参数的普通谓词
- `[terminal]` 匹配字面量，`{code}` 嵌入普通 Prolog
- `phrase/2` 是标准入口，`phrase/3` 可获取剩余
- `listing/1` 看展开是理解 DCG 的最好方式
