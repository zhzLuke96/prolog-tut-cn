> 前置：[元编译器](c5.md) | 难度：★★★ | 后续：—

## 前置知识：Choice Point 与 Cut

### Choice Point

当谓词有多个匹配子句时，Prolog 在第一个子句处设一个选择点（choice point），记录"还有备选"。回溯时引擎回到该点尝试下一条子句。每个 choice point 保存执行状态，过多则拖慢程序。

### Green Cut

Green Cut 是纯逻辑优化——切掉不可能成功的冗余分支，不影响答案集。谓词的逻辑含义不变，只是去掉不必要的回溯。

```prolog
%% Green Cut：子句间互斥
is_digit(X) :- X >= 0, X =< 9, !.
```

如果 X 不满足范围，切掉无意义；满足则提交该子句，避免进入其他分支（这里只有一条子句，! 去除外部 choice point）。

### Red Cut

Red Cut 改变程序语义——删除 cut 后程序逻辑不同，子句顺序变得关键。

```prolog
%% Red Cut：顺序决定语义
max(A, B, A) :- A >= B, !.
max(_, B, B).
```

交换子句顺序会出错：`max(1, 2, M)` 可能返回 M = 1。Red Cut 必须谨慎使用，仅当明确其顺序依赖时。

### Green vs Red 对比

| 对比维度 | Green Cut | Red Cut |
|----------|-----------|---------|
| 删除 cut 影响 | 答案集不变，可能多回溯 | 答案集改变 |
| 安全性 | 安全，纯优化 | 危险，语义依赖 cut |
| 顺序依赖 | 否 | 是 |
| 典型用途 | 提交互斥分支 | 避免不可能路径 |

### 何时用 Cut

优先使用 `(-> ;)/2`（if-then-else），它隐含 cut 且更清晰：

```prolog
%% 推荐：-> ; 代替 cut
max_if(A, B, M) :-
    ( A >= B -> M = A
    ; M = B ).

%% 不推荐：bare cut 容易导致意外语义
max_cut(A, B, A) :- A >= B, !.
```

Bare cut 只在两种情况下合理：1) Green Cut 优化已知互斥的子句；2) 与 fail 配合实现否定。

### 为什么这很重要

理解 choice point 是理解 Prolog 性能的起点。每个 choice point 消耗内存（栈空间）和执行时间（遍历分支）。Green cut 安全地消除冗余 choice point，不改变程序语义；Red cut 改变语义，是最后手段。多数性能问题可以归结为"意外的 choice point"——谓词产生了你没预料到的分支。

经验法则：先用纯逻辑写出正确答案，然后通过 profile 找出冗余 choice point，最后只在热点路径上加 cut。

> 一句话概括：让 Prolog 程序跑得更快、更可预测——逻辑纯度、Tabling、尾递归和 Choice Point 控制。

## 逻辑纯度

逻辑纯度（Logical Purity）指谓词的行为不依赖 Prolog 执行顺序，结果与 Horn 子句的逻辑含义一致。

**纯谓词** 交换子句顺序不影响答案集：

```prolog
%% 纯版本
append([], L, L).
append([H|T], L, [H|R]) :- append(T, L, R).
```

无论调换两个子句顺序，结果一样。

**不纯谓词** 依赖 cut 或 side effect：

```prolog
%% 不纯版本
max(A, B, A) :- A >= B, !.
max(_, B, B).
```

`!/0` 切掉了 choice point，但交换子句顺序会出错。更不纯的例子：

```prolog
avg(L, Avg) :- sum(L, S), length(L, N), Avg is S / N.
```

如果 `sum/2` 或 `length/2` 未实例化参数——`is/2` 要求右边全部实例化——就会抛异常。

保持纯度的方法：

- 避免 cut，用 `if_/3` 或 `-> ;` 替代
- 用 `dif/2` 替代 `\==`
- 把不纯操作（I/O、算术）集中到谓词边界

```prolog
%% 用 dif/2 保持纯度
different(X, Y) :- dif(X, Y).

?- different(a, b).
true .

?- different(a, a).
false .
```

## Tabling / Memoization

SWI-Prolog 的 `library(tabling)` 提供记忆化求值——缓存已计算的子目标结果，避免重复递归。

### 启用方式

```prolog
:- use_module(library(tabling)).
:- table fibonacci/2.
```

参考 `code/hello/fib.pl`（该文件默认用累加器版本，加上 `:- table fib/2.` 即启用 tabling）：

```prolog
:- use_module(library(tabling)).
:- table fib/2.

fib(0, 1) :- !.
fib(1, 1) :- !.
fib(N, F) :-
    N > 1,
    N1 is N - 1,
    N2 is N - 2,
    fib(N1, F1),
    fib(N2, F2),
    F is F1 + F2.
```

没有 tabling 时，`fib(30, _)` 可能要递归几十万次。加上 `:- table fib/2.` 后：

```prolog
?- fib(1000, F).
F = 70330367711422815821835254877183549770181269836358732742604905087154537118196933579742249494562611733487750449241765991088186363265450223647106012053374121273867339111198139373125598767690091902245245323403501.
```

#### 性能对比：fib(30)

```prolog
?- time(fib(30, F)).
% 39,080,737 inferences, 2.500 CPU in 2.500 seconds (100% CPU)
F = 832040.

?- time(fib_tabled(30, F)).
% 182 inferences, 0.000 CPU in 0.000 seconds (100% CPU)
F = 832040.
```

快了几个数量级。Tabling 也解决左递归导致的无限循环问题：

```prolog
:- table ancestor/2.
ancestor(X, Z) :- parent(X, Z).
ancestor(X, Z) :- parent(X, Y), ancestor(Y, Z).
```

> **Prolog 时刻** — tabling 一行 `:- table fib/2.` 将递归调用从指数级降到线性级。声明式优化的终极体现。

### Tabling 限制

- 被 table 的谓词参数必须充分实例化（有限数量的变体）
- 有副作用（assert/retract）的谓词不推荐 tabling
- Tabling 增加内存占用

## 尾递归优化

Prolog 的递归最后一步如果是递归调用，编译器可以复用当前栈帧。对比：

```prolog
%% 非尾递归 — 递归后要执行 is
factorial(0, 1).
factorial(N, F) :-
    N > 0,
    N1 is N - 1,
    factorial(N1, F1),
    F is N * F1.   % 递归后还有工作

%% 尾递归 — 用累加器
factorial_tail(N, F) :-
    factorial_acc(N, 1, F).

factorial_acc(0, Acc, Acc).
factorial_acc(N, Acc, F) :-
    N > 0,
    N1 is N - 1,
    Acc1 is Acc * N,
    factorial_acc(N1, Acc1, F).  % 最后一步是递归
```

尾递归版本在 N 很大时不会栈溢出。检查方法：

```prolog
?- factorial_tail(100000, F).  % 应该成功
```

## Choice Point 消除

Choice point 是 Prolog 回溯时保存的状态点。不必要的 choice point 拖慢程序。

### 实例：成员检查

```prolog
%% 有冗余 choice point
member_of(X, [X|_]).
member_of(X, [_|T]) :- member_of(X, T).

?- member_of(a, [a,b,c]).
true ;
false.   % 回溯后说"没有更多答案"
```

用 `!/0` 或 `once/1` 消除：

```prolog
member_det(X, [X|_]) :- !.
member_det(X, [_|T]) :- member_det(X, T).

?- member_det(a, [a,b,c]).
true .   % 不再有多余 choice point
```

### 实例：color/1 消除 Choice Point

```prolog
%% 有 choice point
color(red).
color(green).
color(blue).

%% 消除后
color_det(red) :- !.
color_det(green) :- !.
color_det(blue) :- !.
```

对比 time/1 输出：

```prolog
?- time((color(_), fail)).
% 9 inferences, 0.000 CPU in 0.000 seconds (100% CPU)
false.

?- time((color_det(_), fail)).
% 3 inferences, 0.000 CPU in 0.000 seconds (100% CPU)
false.
```

### `if_/3` 模式

SWI-Prolog 的 `library(reif)` 提供可重用的纯度模式：

```prolog
:- use_module(library(reif)).

%% 纯的"如果-否则"
cond_if(P, Then, Else) :-
    if_(P, Then, Else).
```

但日常更常用 `-> ;`：

```prolog
%% 用 -> ; 替代 cut
max_cut(A, B, A) :- A >= B, !.
max_cut(_, B, B).

max_if(A, B, M) :-
    (   A >= B
    ->  M = A
    ;   M = B
    ).
```

### 何时不该消除

- 需要回溯枚举所有答案时不能消除
- 不确定是否有 choice point 时，用 `predicate_property/2` 检查

```prolog
?- predicate_property(append([], _, _), P).
P = visible ;
P = interpreted ;
P = nondeterministic ;  % 非确定性 = 有 choice point
...
```

## 综合建议

1. 优先写纯逻辑，再按需加 cut 优化
2. Fibonacci 类重复计算问题 → Tabling
3. 线性递归且最后一步是自身 → 尾递归改写
4. 确定只需一个答案且 cut 不影响逻辑 → `!/0` 或 `once/1`
5. 用 `library(reif)` 写纯条件
6. 性能分析用 `time/1` 和 `profile/1`

```prolog
?- time(fib(30, _)).
% 40,317,617 inferences, 2.344 CPU in 2.344 seconds
```

> 参考：[The Power of Prolog - Purity](https://www.metalevel.at/prolog/purity) — 关于逻辑纯度的深度讨论。

## 总结

优化不是"写得更快"，而是"让引擎做更少的事"。逻辑纯度保证程序语义不受优化破坏；Tabling 用空间换时间；尾递归压缩栈空间；choice point 消除减少回溯浪费。四者结合，Prolog 程序能从"玩具"变成"实用"。
